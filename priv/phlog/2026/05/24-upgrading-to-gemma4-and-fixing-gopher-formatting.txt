Upgrading to Gemma 4 and Fixing Gopher Formatting

It has been a busy day at GopherLab. We tackled a stack of issues
that had been keeping the server down and made some major upgrades
to the AI backend. Here is a rundown of everything that changed.

=== The Server Was Down ===

The server had been crash-looping on startup. The root cause turned
out to be an architecture mismatch in the EXLA native library. The
compiled libexla.so was built for x86_64 (Intel), but the server
runs on Apple Silicon (arm64). Every time the application tried to
load the NIF, it would fail immediately:

  "mach-o file, but is an incompatible architecture
   (have 'x86_64', need 'arm64')"

This likely happened from a previous build under Rosetta emulation.
The fix was straightforward: clean the old build artifacts and
recompile EXLA for the correct architecture.

On top of that, Xcode 16.3 introduced a breaking change in its
C++ standard library headers. The XLA extension headers specialize
std::is_signed which newer clang now treats as a hard error. We
worked around this by passing a compiler flag:

  CFLAGS="-Wno-error=invalid-specialization"

This has been baked into the sync-gopher.sh deployment script so
future deploys will not hit the same issue.

=== Switching to Ollama with Gemma 4 ===

The previous AI setup used Bumblebee with a Llama 3.2 1B model
loaded directly from HuggingFace. This had two problems:

  1. Llama 3.2 is a gated model requiring an HF token
  2. The gopher service user did not have the token configured

We briefly tried switching back to DeepSeek R1 Distill (which was
the model before Llama 3.2), but Bumblebee 0.6.3 does not support
the Qwen2 architecture that model uses.

Rather than fight Bumblebee compatibility, we switched the primary
AI backend to Ollama. The server already had full Ollama integration
built in -- it just was not enabled by default.

The new setup:

  Primary:  Ollama with gemma4:e2b (Google Gemma 4, ~7.2GB)
  Fallback: Bumblebee with GPT-2 (lightweight, always loaded)

Gemma 4 E2B is the instruction-tuned variant with support for chat,
tool use, and thinking capabilities. Response times are solid:

  Cold start:  ~3 seconds (model loading into GPU memory)
  Warmed up:   ~1-2 seconds

We initially tried the larger E4B model but it was too slow for
Gopher clients which have short connection timeouts. The E2B model
hits a good balance of quality and speed.

For the record, we also checked whether upgrading Bumblebee would
help. The latest version (0.7.0) adds Gemma 3 text-only support
but Gemma 4 is not supported in any version. A community PR to add
it was opened and closed without merging.

=== Fixing the Gopher Response Formatting ===

After getting the server back up, AI responses were rendering
incorrectly in Gopher clients. Instead of readable text, users
would see rows of document icons with no links and a black
background. Not a great experience.

The problem was in how streamed AI responses were formatted. The
/ask and /chat selectors are Gopher type 7 (search/query). When a
client receives a response from a type 7 selector, it expects each
line to be a proper Gopher menu item:

  i<text>\t\t<host>\t<port>\r\n

But the streaming code was sending raw plain text. Characters at
the start of lines like Q (Query:), R (Response:), and G
(Generated) were being interpreted as Gopher item type codes.

The fix was switching all six AI streaming functions from
start_plain_buffered_streamer (raw text) to start_buffered_streamer
(proper Gopher i-line formatting). Headers and footers were also
updated to go through the Gopher formatting pipeline.

We also added navigation links at the bottom of every AI response:

  - Ask Another Question  (/ask)
  - Chat with AI          (/chat)
  - Back to Main Menu     (/)

Previously, AI responses were dead ends with no way to navigate
back without manually editing the URL.

=== What Still Needs to Be Done ===

There are a few things on the radar:

  Upgrade Bumblebee and EXLA
    We are on Bumblebee 0.6.3 and EXLA 0.10.0. The latest versions
    are 0.7.0 and 0.12.0 respectively. Upgrading would bring Gemma
    3 text support for Bumblebee and might fix the Xcode compile
    issue for EXLA. This is a bigger change that needs testing.

  Ollama Keep-Alive
    The first request after Ollama unloads the model from memory
    takes a few seconds. We could configure Ollama's keep_alive
    setting to keep the model loaded longer, or send periodic
    warmup requests.

  Better Error Handling for Ollama
    If Ollama is not running, the server falls back to GPT-2 via
    Bumblebee. The fallback works, but the user gets no indication
    that they are talking to a much smaller model. We should surface
    which model generated the response.

  Gemini Protocol Testing
    The Gemini handler has not been tested with the new Ollama
    backend. It should work since the AI engine abstraction handles
    the backend selection, but it needs verification.

  Model Display in Responses
    It would be nice to show which model is responding (gemma4:e2b
    vs gpt2) in the response footer so users know what they are
    getting.

=== Configuration Reference ===

For anyone running their own instance, here are the key environment
variables for the new setup:

  OLLAMA_ENABLED=true         Enable Ollama (default: true)
  OLLAMA_URL=http://...       Ollama server URL (default: localhost:11434)
  OLLAMA_MODEL=gemma4:e2b     Model to use (default: gemma4:e2b)
  AI_MODEL=openai-community/gpt2  Bumblebee fallback model

Make sure Ollama is running before starting the server:

  ollama serve
  ollama pull gemma4:e2b
